The problem

Building a production app means stitching together a dozen tools that don't talk to each other.

Your ORM doesn't know about your API schema. Your API schema doesn't know about your frontend types. Your frontend types don't know about your validation rules. Every seam is a place where bugs hide and LLMs guess wrong.

We're eliminating the seams.

The vision

Vertz is not a framework.
It's not a piece of the stack.
It's the stack.

One type system. One set of conventions. One source of truth that flows through every layer. When you define a schema, it becomes your database table, your API contract, your client types, and your form validation — automatically, correctly, and with zero manual wiring.

This is not a 5-year dream. This is the roadmap. We ship it piece by piece, and every piece we ship is the best-in-class tool for that layer.

How we build

If it builds, it works

The compiler is the quality gate — not your eyes, not manual testing, not hoping the runtime resolves correctly. We push everything to compile time so TypeScript catches it before you deploy.

AI agents are first-class users

Every API decision is evaluated by one question: can an LLM use this correctly on the first prompt? Designing for AI makes us design better for everyone.

No ceilings

If a dependency limits us, we replace it. If the runtime is too slow, we build a faster one. The vision determines the tools, not the other way around.

What it looks like

app.ts 
// Define your schema once
const User = s.object({
  name: s.string(),
  email: s.string().email(),
  role: s.enum(['admin', 'member']),
})

// Types flow to your database
const users = await db.user.findMany()
//    ^? User[] — fully typed, zero codegen

// Types flow to your API
router.get('/', {
  response: s.array(User),
  handler: () => db.user.findMany(),
})

// Types flow to your client — no manual annotations
const { data } = client.users.list()
//      ^? User[] — inferred from the schema

The stack

Schema — 40+ types with runtime validation, type inference, and JSON Schema output. One definition powers everything.

Database — an ORM that speaks the same schema language. Same types from table to API. No translation layer.

API — modules, services, routers, middleware, DI, OpenAPI generation. No decorators. Functions compose. The compiler catches the rest.

Client — generated SDK with full type safety. If your API changes, your client knows at compile time.

UI — components that consume your schema types directly. From database to browser, one source of truth.
