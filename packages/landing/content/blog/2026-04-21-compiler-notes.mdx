## Why write a loader

Every framework's MDX integration is a compromise. Each one couples the
parser, the JSX runtime, and the deployment target in a way that's hard to
swap out later. `@vertz/mdx` keeps the unified pipeline configurable so a
blog, a docs site, or a cookbook can all plug into the same parser with
different runtimes.

## The pipeline in one picture

1. `@mdx-js/mdx` parses the source into a hast tree.
2. `@shikijs/rehype` syntax-highlights fenced code blocks.
3. A rehype slug injector assigns stable ids to h2 / h3.
4. A tiny JSX shim serializes the tree into an HTML string.

## What we gave up

- No dynamic component overrides at the MDX layer (deferred to Phase 4).
- The body renders through `innerHTML`, not JSX — hydration is static.

## What we kept

`@vertz/mdx`'s `compileMdx()` remains the single source of truth for
frontmatter parsing. The blog just adds a second pass to produce the
string output.
