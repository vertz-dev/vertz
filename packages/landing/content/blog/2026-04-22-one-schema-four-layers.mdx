## The four-copies problem

Ship a feature in a "modern" stack and you end up writing the same shape four times.

First in the migration — a SQL column list, probably generated by Prisma or a hand-written `CREATE TABLE`. Second in the server — a DTO, a validator, a response schema, something. Third in the client — whatever your REST/GraphQL codegen spits out, usually with `| null` in all the wrong places. Fourth in the form — Zod or Yup or Joi, with its own types that subtly disagree with the DTO you wrote an hour ago.

Every copy is a seam. Every seam is a place where the name, the nullability, or the type drifts. Every drift is a bug that only surfaces when a real user hits the endpoint with a value your frontend didn't expect. You wrote the same thing four times and the compiler only checked that each copy was internally consistent.

We decided Vertz would not have that shape.

## One definition

A table in Vertz is a function call. The `d` builder from `@vertz/db` is the entire schema API — no separate `.prisma` file, no decorators, no code generation you have to remember to run.

```ts
import { d } from '@vertz/db';

const tasks = d.table('tasks', {
  id: d.uuid().primary({ generate: 'cuid' }),
  title: d.text(),
  status: d.enum('task_status', ['todo', 'in_progress', 'done']).default('todo'),
  assigneeId: d.uuid().nullable(),
  createdAt: d.timestamp().default('now').readOnly(),
});
```

That's not just a database table. That's the source for four layers.

### Layer 1 — the database

Column builders map to real SQL. `d.uuid().primary({ generate: 'cuid' })` generates a `UUID PRIMARY KEY` with CUID2 defaults at write time. `d.enum(...)` creates the `task_status` enum type. `d.timestamp().default('now')` emits `TIMESTAMPTZ DEFAULT NOW()`. Foreign keys are derived from relations, not declared twice — if you add `d.ref.one(() => usersTable, 'assigneeId')`, the FK constraint lands in the migration automatically.

Postgres, SQLite, and Cloudflare D1 all read the same definition. Dialect-specific features (GIN indexes, pgvector) validate at definition time — a `type: 'gin'` index against SQLite warns before migration, not after it blows up with a cryptic SQL error.

### Layer 2 — the API server

Wrap the table in a model, wrap the model in an entity, and you have a REST API:

```ts
import { entity, rules } from '@vertz/server';

const tasksModel = d.model(tasks);

const tasksEntity = entity('tasks', {
  model: tasksModel,
  access: {
    list: rules.authenticated(),
    get: rules.authenticated(),
    create: rules.authenticated(),
  },
});
```

The framework derives three different types from the same columns:

| Type                | Includes                   | Excludes                                                            |
| ------------------- | -------------------------- | ------------------------------------------------------------------- |
| `$response`         | `id`, `title`, `status`, `createdAt` | Hidden fields                                            |
| `$create_input`     | `title`, `status` (optional), `assigneeId` | `id` (primary), `createdAt` (readOnly)           |
| `$update_input`     | All of the above as optional | Same exclusions                                                   |

You never write these manually. `.readOnly()` on `createdAt` is what removes it from create and update inputs. `.hidden()` on a `passwordHash` column would remove it from responses. The annotations you wrote on the column are the ones that drive the API surface.

The routes — `GET /api/tasks`, `POST /api/tasks`, `PATCH /api/tasks/:id` — are generated from the `access` object. If you don't define an access rule for `delete`, the route doesn't exist. No 404, no 405, nothing to discover. Deny-by-default is the same principle in a different layer.

### Layer 3 — the client SDK

When you run `vtz dev`, the compiler reads every entity you defined and emits a typed client. No separate codegen step, no `openapi-typescript-codegen`, no "remember to regenerate after changing the schema."

```ts
const result = await api.tasks.list({
  where: { status: 'todo' },
  orderBy: { createdAt: 'desc' },
});
// result.items[0].status — typed as 'todo' | 'in_progress' | 'done'
// result.items[0].createdAt — typed as Date
```

The types on `api.tasks.list` are the same types derived from the same `d.table()` call. Change the enum values in the schema and the client's `.status` field tightens immediately. The compiler catches the mismatch in the component using it, not in QA a week later.

### Layer 4 — the UI forms

The entity's `$create_input` type is the shape a form needs. The framework's `form()` helper takes that same schema and drives client-side validation:

```ts
import { form } from '@vertz/ui';

const createTask = form(api.tasks.create, {
  onSuccess: () => navigate({ to: '/tasks' }),
});
```

Field errors, submit states, and the HTTP method come from the entity. The form knows `title` is required because the column isn't `.nullable()`. It knows `status` is optional because the column has `.default('todo')`. It rejects `id` in the submit payload because the column is `.primary()`. No one wrote this logic twice.

## What we rejected

**Prisma-style post-hoc codegen.** The schema lives in a `.prisma` file, you run `prisma generate`, and your TypeScript gets types. It works, but the generation step is a second source of truth that falls out of sync whenever someone forgets to run it. We wanted the schema to be TypeScript, not to produce TypeScript.

**Manual DTOs with class-validator.** The NestJS shape. You write a class, decorate it, and hope the decorators match the ORM entity and the frontend interface. Types don't flow through decorator chains — the compiler can't help you, and LLMs get it wrong constantly. It's the decorator-heavy pattern we built Vertz specifically to replace.

**GraphQL codegen.** Best-in-class tooling for the four-copies problem and still not good enough: you're generating a schema language from TypeScript, then generating TypeScript from the schema language. Every transition is a chance to lose nullability, date handling, or enum narrowing.

## What this costs you

Honesty time. This design has tradeoffs.

You're locked into the Vertz type system. The one-schema story works because every layer speaks TypeScript and reads the same `d.table()` object. If you want a Python client, a Rust consumer, or a Swift mobile app, you're on your own — you can hit the auto-generated OpenAPI, but none of the ergonomics we just described come with it. The story is "TypeScript end-to-end," not "polyglot end-to-end."

You also give up the escape hatch of "just hand-write this one DTO differently." The column annotations drive the derived types. If you need a response shape that diverges from what `.hidden()` and `.readOnly()` express, you use `expose` to narrow the entity's surface — but you can't widen it past the schema. The schema is the contract.

We think that's the right tradeoff. The cost of the four-copies problem is paid every day by every developer on the team. The cost of being TypeScript-first is paid once, at stack-choice time, by the architect.
