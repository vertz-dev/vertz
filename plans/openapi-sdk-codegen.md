# OpenAPI SDK Code Generator

> **Rev 2** — Addresses all findings from DX, Product/Scope, and Technical reviews.

## Summary

A standalone tool that generates typed TypeScript SDKs from any OpenAPI 3.x specification. Targets the broader community — any backend framework that produces OpenAPI (FastAPI, NestJS, Rails, Django, Go, etc.) gets a type-safe TypeScript client for free.

**Package:** `@vertz/openapi`
**CLI:** `npx @vertz/openapi generate --from openapi.json`

---

## Why Now?

Vertz is pre-v1 with active work on runtime, test runner, and agent orchestrator. This tool is **not on the critical path** for any of those. It's a standalone package with zero Vertz runtime dependencies — a different agent can build it in parallel without blocking or being blocked by core initiatives.

Strategic value:
1. **Community on-ramp.** `@vertz/openapi` is the first Vertz package most developers will encounter. It works with their existing stack (FastAPI + React, NestJS + Vue, etc.) and requires zero buy-in to the Vertz framework.
2. **Proves the codegen architecture.** Building a standalone generator exercises the same patterns the internal `@vertz/codegen` uses, surfacing design gaps (like the `CodegenResolvedField` type narrowness found during this review).
3. **Low-effort, high-visibility.** Phases 1-3 are ~2-3 days of agent work. The output is a publishable npm package with immediate utility.
4. **Build-in-public content.** A community tool generates social proof and Twitter content.

**Timing:** Ship after the current sprint's active work (no interruption), built by a dedicated agent in a worktree.

**client.ts:**
```typescript
// Generated by @vertz/openapi — do not edit

export interface HttpClient {
  get<T>(path: string, options?: { query?: Record<string, unknown> }): Promise<T>;
  post<T>(path: string, body?: unknown): Promise<T>;
  put<T>(path: string, body?: unknown): Promise<T>;
  patch<T>(path: string, body?: unknown): Promise<T>;
  delete<T>(path: string): Promise<T>;
}

export interface ClientOptions {
  baseURL?: string;
  headers?: Record<string, string>;
  fetch?: typeof globalThis.fetch;
}

import { createTasksResource } from './resources/tasks';
import { createUsersResource } from './resources/users';

export function createClient(options: ClientOptions = {}) {
  const baseURL = (options.baseURL ?? '').replace(/\/$/, '');
  const customFetch = options.fetch ?? globalThis.fetch.bind(globalThis);
  const defaultHeaders = options.headers ?? {};

  const client: HttpClient = {
    async get<T>(path: string, opts?: { query?: Record<string, unknown> }): Promise<T> {
      let url = `${baseURL}${path}`;
      if (opts?.query) {
        const params = new URLSearchParams();
        for (const [k, v] of Object.entries(opts.query)) {
          if (v !== undefined && v !== null) params.set(k, String(v));
        }
        const qs = params.toString();
        if (qs) url += `?${qs}`;
      }
      const res = await customFetch(url, { method: 'GET', headers: defaultHeaders });
      if (!res.ok) throw await ApiError.from(res);
      if (res.status === 204) return undefined as T;
      return res.json();
    },

    async post<T>(path: string, body?: unknown): Promise<T> {
      const res = await customFetch(`${baseURL}${path}`, {
        method: 'POST',
        headers: { ...defaultHeaders, 'Content-Type': 'application/json' },
        body: body !== undefined ? JSON.stringify(body) : undefined,
      });
      if (!res.ok) throw await ApiError.from(res);
      if (res.status === 204) return undefined as T;
      return res.json();
    },

    // put, patch follow same pattern as post
    // delete follows same pattern but without body

    async put<T>(path: string, body?: unknown): Promise<T> {
      const res = await customFetch(`${baseURL}${path}`, {
        method: 'PUT',
        headers: { ...defaultHeaders, 'Content-Type': 'application/json' },
        body: body !== undefined ? JSON.stringify(body) : undefined,
      });
      if (!res.ok) throw await ApiError.from(res);
      if (res.status === 204) return undefined as T;
      return res.json();
    },

    async patch<T>(path: string, body?: unknown): Promise<T> {
      const res = await customFetch(`${baseURL}${path}`, {
        method: 'PATCH',
        headers: { ...defaultHeaders, 'Content-Type': 'application/json' },
        body: body !== undefined ? JSON.stringify(body) : undefined,
      });
      if (!res.ok) throw await ApiError.from(res);
      if (res.status === 204) return undefined as T;
      return res.json();
    },

    async delete<T>(path: string): Promise<T> {
      const res = await customFetch(`${baseURL}${path}`, {
        method: 'DELETE',
        headers: defaultHeaders,
      });
      if (!res.ok) throw await ApiError.from(res);
      if (res.status === 204) return undefined as T;
      return res.json();
    },
  };

  return {
    tasks: createTasksResource(client),
    users: createUsersResource(client),
  };
}

export class ApiError extends Error {
  override name = 'ApiError';
  public data: unknown;

  constructor(public status: number, body: string) {
    super(`API error ${status}: ${body}`);
    try { this.data = JSON.parse(body); } catch { this.data = body; }
  }

  static async from(res: Response): Promise<ApiError> {
    const body = await res.text();
    return new ApiError(res.status, body);
  }
}

export type Client = ReturnType<typeof createClient>;
```

## Manifesto Alignment

### Type Safety Wins
OpenAPI specs define request/response schemas as JSON Schema. We convert these to TypeScript interfaces (and optionally Zod runtime schemas). The generated SDK is fully typed — if it compiles, the shapes match the API contract.

### One Way to Do Things
One tool, one command, one output structure. The generated code looks like hand-written TypeScript — clean, readable, predictable. No `--name`, `--style`, or `--template` knobs.

### AI Agents Are First-Class Users
The generated SDK has exactly one way to call each endpoint. An LLM can look at `api.tasks.create({ title: 'New' })` and immediately know what it does. No ambiguity, no overloaded methods, no config objects with 12 optional fields.

### If It Builds, It Works
Types are generated directly from the OpenAPI spec. If the spec changes, the generated types change, and TypeScript catches every callsite that broke. No runtime surprises.

### Tradeoff: Convention Over Configuration
We choose sensible defaults (group by tag, zero dependencies, use plain fetch) rather than exposing every knob. Power users can switch to config file mode.

---

## Non-Goals

1. **Not a full OpenAPI client generator** — We don't generate API classes, interceptors, auth middleware, or mock servers. We generate typed SDK methods.
2. **No support for OpenAPI 2.0 (Swagger)** — Only OpenAPI 3.0 and 3.1. Swagger 2.0 is legacy; converters exist.
3. **No code-first generation** — We don't analyze Python/Go/Java source code. We consume the spec.
4. **No server stub generation** — We only generate TypeScript clients, not server implementations.
5. **No GraphQL** — OpenAPI (REST) only. GraphQL is a different paradigm.
6. **No framework-specific hooks** — We don't generate React hooks, Vue composables, or Svelte stores. We generate plain functions that any framework can wrap.
7. **No runtime validation on API calls** — Schemas (when enabled) are for user-side validation (forms, inputs), not for validating every API response.
8. **No auth helpers** — Security schemes in the spec are not turned into auth middleware. Users handle auth via the `headers` or `fetch` options.
9. **No typed error responses (v1)** — Generating typed `4xx`/`5xx` error schemas is deferred. `ApiError.data` provides the raw parsed body for now.
10. **No external `$ref`s** — Multi-file specs (`$ref: './models/user.yaml'`) are not supported in v1. Single-file specs only. A clear error is shown for external refs.
11. **No Vertz integration (v1)** — `runtime: 'vertz'` mode (descriptor-based SDK, optimistic updates) is deferred to a separate design doc after `@vertz/codegen` stabilizes. This design covers the standalone community tool only.

## Unknowns

### 1. Resource Grouping Strategy
**Question:** How should we group OpenAPI paths into "resources" (entity-like groupings)?

**Resolution:** `tag` is the default. Operations with no tags go into an `_ungrouped` resource. Operations with multiple tags use the first tag. Tag names are sanitized to valid TypeScript identifiers (`Task Management` → `taskManagement`, `v2/tasks` → `v2Tasks`). `path` mode strips common prefixes (`/api/`, version segments like `/v2/`).

### 2. Schema Library
**Question:** Should generated schemas use Zod or something else?

**Resolution:** Zod — it's the industry standard with 50M+ weekly downloads. `schemas: false` is the default (zero dependencies). When `schemas: true`, `zod` becomes a peer dependency.

### 3. Complex JSON Schema Features
**Question:** How do we handle `allOf`, `oneOf`, `anyOf`, `discriminator`?

**Resolution:** Phase 1 handles `$ref`, basic `allOf` (merge), `nullable`, arrays, and enums. Phase 2 (follow-up) adds `oneOf`/`anyOf` → TypeScript union types. `discriminator` maps to tagged unions in Phase 2.

### 4. OpenAPI 3.0 vs 3.1 Differences
**Question:** How do we handle the schema differences between spec versions?

**Resolution:** Detect version from the `openapi` field. Key differences handled:
- `nullable: true` (3.0) vs `type: ['string', 'null']` (3.1) → both emit `T | null`
- `exclusiveMinimum` as boolean (3.0) vs number (3.1)
- `$ref` with siblings (3.1 allows, 3.0 does not)

## Architecture

### Why Not Reuse `CodegenIR`?

The existing `CodegenIR` and its `CodegenResolvedField` type only support 5 primitive types (`string | number | boolean | date | unknown`). OpenAPI schemas produce arrays, nested objects, enums, unions, nullable types, and `$ref` references — all of which would degrade to `unknown` through `CodegenResolvedField`.

Instead, the standalone generators work directly with parsed JSON Schema objects and convert them to TypeScript strings using a dedicated `jsonSchemaToTS()` function. This produces correct TypeScript for the full JSON Schema type space.

Future Vertz integration (`runtime: 'vertz'`) will be addressed in a separate design doc, potentially involving an adapter from OpenAPI's richer schema model to `CodegenIR` with documented limitations.

### Internal Type Representation

```typescript
/** Parsed and resolved OpenAPI operation */
interface ParsedOperation {
  operationId: string;
  method: HttpMethod;
  path: string;
  tags: string[];
  pathParams: ParsedParameter[];
  queryParams: ParsedParameter[];
  requestBody?: ParsedSchema;     // Resolved JSON Schema
  response?: ParsedSchema;        // Resolved JSON Schema (success response)
  responseStatus: number;         // 200, 201, 204, etc.
}

interface ParsedParameter {
  name: string;
  required: boolean;
  schema: ParsedSchema;
}

/** JSON Schema with $refs fully resolved */
interface ParsedSchema {
  /** Original schema name from components/schemas (if any) */
  name?: string;
  /** Fully resolved JSON Schema object (no $refs) */
  jsonSchema: Record<string, unknown>;
}

/** A resource is a group of operations sharing a tag or path prefix */
interface ParsedResource {
  name: string;           // e.g., 'tasks'
  identifier: string;     // e.g., 'tasks' (valid TS identifier)
  operations: ParsedOperation[];
}
```

### Operation ID Normalization

Frameworks generate inconsistent operation IDs. The generator normalizes them in three steps:

**Step 1: Auto-clean** (always runs)
- Strip common framework prefixes: `TasksController_findAll` → `findAll`
- Strip redundant path segments: `list_tasks_tasks__get` → `listTasks`
- Normalize to camelCase: `get-task-by-id` → `getTaskById`
- Strip HTTP method suffixes: `listTasksGet` → `listTasks`

**Step 2: Transform function** (optional, from config)
```typescript
// Example: strip 'Api' prefix from all IDs
transform: (cleaned, _original) => cleaned.replace(/^api/, ''),
```

**Step 3: Static overrides** (optional, from config)
```typescript
overrides: { 'list_tasks_tasks__get': 'list' }
```

**Step 4: CRUD method detection**
When an operation maps to a standard CRUD pattern (based on HTTP method + path shape), the method name is simplified:
- `GET /tasks` → `list` (regardless of original operationId)
- `GET /tasks/{id}` → `get`
- `POST /tasks` → `create`
- `PATCH/PUT /tasks/{id}` → `update`
- `DELETE /tasks/{id}` → `delete`

Non-CRUD operations keep their normalized operationId as the method name.

This means developers get clean `api.tasks.list()` even when FastAPI generates `list_tasks_tasks__get`.

### `$ref` Resolution Strategy

1. **Internal `$ref`s** (`#/components/schemas/Task`): Resolved by walking the document tree. Named schemas produce named TypeScript interfaces/Zod schemas. Inline schemas produce anonymous types.
2. **Circular `$ref`s**: Detected via a `resolving: Set<string>` guard. In TypeScript output, circular refs emit the named type as a forward reference. In Zod output, circular refs emit `z.lazy(() => schemaRef)`.
3. **Deep `$ref` chains**: Iteratively resolved until a non-`$ref` schema is reached, with cycle detection.
4. **External `$ref`s**: Not supported in v1. The parser throws a clear error: `External $ref not supported: ${ref}. Inline the schema or use a tool like swagger-cli to bundle the spec.`
5. **`$ref` + siblings** (3.1 only): Sibling keywords are merged with the resolved schema. In 3.0 mode, siblings next to `$ref` are ignored per the spec.

## Type Flow Map

```
OpenAPI 3.x Spec (JSON)
  │
  ├─ paths['/tasks'].get.responses['200'].content['application/json'].schema
  │    ↓
  │  JSON Schema { type: 'object', properties: { id: { type: 'string' }, tags: { type: 'array', items: { type: 'string' } } } }
  │    ↓
  │  jsonSchemaToTS() → 'Task' (named interface)
  │    ↓
  │  TypeScript interface: `export interface Task { id: string; tags: string[]; ... }`
  │    ↓
  │  Resource method return type: `get(taskId: string): Promise<Task>`
  │    ↓
  │  User code: `const task = await api.tasks.get('123');`
  │              //    ^? Task — fully typed, including arrays, enums, nested objects
  │
  ├─ paths['/tasks'].get.parameters[?in='query']
  │    ↓
  │  Parameter schemas → `ListTasksQuery` interface
  │    ↓
  │  Resource method param type: `list(query?: ListTasksQuery): Promise<Task[]>`
  │    ↓
  │  User code: `await api.tasks.list({ status: 'active', limit: 10 });`
  │              //                      ^? { status?: string; limit?: number; ... }
  │
  ├─ paths['/tasks'].post.requestBody.content['application/json'].schema
  │    ↓
  │  JSON Schema → jsonSchemaToTS() → TypeScript interface: `CreateTaskInput`
  │    ↓
  │  Resource method param type: `create(body: CreateTaskInput): Promise<Task>`
  │    ↓
  │  User code: `await api.tasks.create({ title: 'New' });`
  │              //                        ^? { title: string; done?: boolean; tags?: string[] }
  │
  └─ components.schemas['Task']  (when schemas: true)
       ↓
     jsonSchemaToZod() → Zod expression string
       ↓
     `export const taskSchema = z.object({ id: z.string().uuid(), tags: z.array(z.string()), ... })`
       ↓
     User code: `taskSchema.parse(data)` — runtime validation
```

**No dead generics.** Every type flows from the JSON Schema in the OpenAPI spec through to the user's variable. The `<T>` in `client.get<T>()` is always resolved to a concrete interface derived from the spec.

## Future Work (deferred)

These are explicitly out of scope for v1 but tracked for future design docs:

1. **Watch mode** — File watcher for local specs, URL polling for remote specs. Deferred because manual re-run or build-script integration covers the 90% case.
2. **Vertz integration** (`runtime: 'vertz'`) — Descriptor-based SDK output using `@vertz/fetch`. Requires a separate design doc addressing how OpenAPI's richer schema model maps to `CodegenIR` limitations.
3. **Framework plugins** — FastAPI middleware, NestJS decorator that auto-export the spec on change. Depends on community adoption and feedback.
4. **`oneOf`/`anyOf` unions** — TypeScript discriminated union types from `discriminator` + `oneOf`.
5. **Typed error responses** — Generate typed error types from `4xx`/`5xx` response schemas.
6. **Multiple schema libraries** — Valibot, ArkType output (alongside Zod).

## Dependencies

The `@vertz/openapi` package has **zero mandatory runtime dependencies**:
- Generated code uses the built-in `fetch` API — no HTTP client library
- TypeScript interfaces are zero-cost (erased at runtime)

**Optional peer dependency:**
- `zod` — only when `schemas: true` is enabled in config

**Dev/build dependencies:**
- `yaml` — for parsing YAML spec files (bundled into the CLI, not a user dependency)
- Standard Node.js built-ins (`fs`, `path`)

## Known Limitations (v1)

1. **JSON only for request bodies** — The generated client always sends `application/json`. `multipart/form-data`, `application/x-www-form-urlencoded`, and binary uploads are not supported. Operations with non-JSON request bodies generate a method with `body: unknown` and a `// TODO: non-JSON content type` comment.
2. **Simple query serialization** — Query parameters are serialized with `URLSearchParams` (key=value). OpenAPI's `style: 'deepObject'` and array serialization styles are not supported.
3. **Single-file specs only** — External `$ref`s to other files are not resolved. Use `swagger-cli bundle` to inline them first.
4. **No per-request headers** — Headers are set globally via `ClientOptions.headers`. Per-request headers (auth tokens, idempotency keys) should be handled via the `fetch` option (custom fetch wrapper).

## Package Name Decision

**Chosen: `@vertz/openapi`**

Rationale: The `@vertz` scope signals quality and a curated ecosystem. The subcommand (`generate`) makes the purpose clear in CLI usage. Developers discovering `@vertz/openapi` through npm search will find the Vertz ecosystem — which is the point. Brand association with a pre-v1 framework is acceptable because this tool works standalone and doesn't require any other `@vertz/*` packages.

If community feedback indicates the `@vertz` scope hurts adoption, we can publish under a standalone name and make `@vertz/openapi` an alias.
