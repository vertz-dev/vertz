## 1. Problem Statement

`defineAccess()` is fully implemented at runtime — entitlements, roles, hierarchy, plans, and wallet all resolve correctly. But `ctx.can('project:fly')` is not a compile error. It's a runtime bug that resolves to `false` and silently denies access.

The compiler already extracts entity definitions, schema types, and route information from source code. It should also extract access metadata so that:

- `ctx.can('typo')` is a **compile error**, not a silent denial
- `rules.role('nonexistent')` is caught by TypeScript
- Autocomplete works for entitlement names in both server and client code

This is Vertz Principle #1: **"If it builds, it works."** Authorization strings are a gap in that guarantee.

## 2. Architecture Overview

```
User code                     Compiler                    Codegen
─────────                     ────────                    ───────

defineAccess({          ┌─ AccessAnalyzer ──┐     ┌─ AccessTypesGenerator ─┐
  entities: {           │                   │     │                        │
    project: {          │  Extract string   │     │  Emit .d.ts with       │
      roles: [...]      │  literals from    │     │  module augmentations  │
    }                   │  AST config obj   │     │  for @vertz/server     │
  },                    │         │         │     │  and @vertz/ui         │
  entitlements: {       │         ▼         │     │                        │
    'project:view': ... │    AccessIR       │────▶│  Entitlement union     │
    'project:edit': ... │  { entitlements,  │     │  ResourceType union    │
  }                     │    entities,      │     │  Role<T> conditional   │
})                      │    roles }        │     └────────────────────────┘
                        └──────────────────┘

                                                  ┌─ RlsPolicyGenerator ──┐
rules.where({           ┌─ AccessAnalyzer ──┐     │                       │
  createdBy:            │  Extract where    │────▶│  CREATE POLICY ...    │
    rules.user.id       │  conditions as    │     │  USING (...)          │
})                      │  structured data  │     └───────────────────────┘
                        └──────────────────┘
```

**Key constraint:** The compiler is per-file, single-pass. `defineAccess()` is typically a single call in a single file (e.g., `src/access.ts`). The analyzer extracts string literals from the AST — no cross-file analysis needed.

## 3. API Surface

### 3.1 Input: What the user writes (no changes)

```ts
import { defineAccess, rules } from '@vertz/server';

export const access = defineAccess({
  entities: {
    workspace: {
      roles: ['admin', 'member'],
    },
    project: {
      roles: ['manager', 'contributor', 'viewer'],
      inherits: { 'workspace:admin': 'manager', 'workspace:member': 'viewer' },
    },
  },
  entitlements: {
    'project:view': { roles: ['viewer', 'contributor', 'manager'] },
    'project:edit': { roles: ['contributor', 'manager'] },
    'project:delete': (r) => ({
      roles: ['manager'],
      rules: [r.where({ createdBy: r.user.id })],
    }),
    'workspace:invite': { roles: ['admin'] },
  },
});
```

### 3.2 Output: What the codegen generates

**File: `.vertz/generated/access.d.ts`**

```ts
// Generated by @vertz/codegen — do not edit

declare module '@vertz/server' {
  interface AccessContext {
    can(entitlement: Entitlement, resource?: ResourceRef): Promise<boolean>;
    check(entitlement: Entitlement, resource?: ResourceRef): Promise<AccessCheckResult>;
    authorize(entitlement: Entitlement, resource?: ResourceRef): Promise<void>;
  }
}

declare module '@vertz/ui' {
  export function can(
    entitlement: Entitlement,
    entity?: { __access?: Record<string, import('@vertz/ui').AccessCheckData> },
  ): import('@vertz/ui').AccessCheck;
}

declare module '@vertz/ui/auth' {
  export function can(
    entitlement: Entitlement,
    entity?: { __access?: Record<string, import('@vertz/ui').AccessCheckData> },
  ): import('@vertz/ui').AccessCheck;
}

type Entitlement =
  | 'project:view'
  | 'project:edit'
  | 'project:delete'
  | 'workspace:invite';

type ResourceType = 'workspace' | 'project';

type Role<T extends ResourceType> =
  T extends 'workspace' ? 'admin' | 'member' :
  T extends 'project' ? 'manager' | 'contributor' | 'viewer' :
  never;
```

### 3.3 Developer experience after codegen

```ts
// Server — autocomplete + compile error on typo
ctx.can('project:view');          // OK
ctx.can('project:edit');          // OK

// @ts-expect-error — 'project:fly' is not a valid Entitlement
ctx.can('project:fly');

// Client — same type safety
const canEdit = can('project:edit');
canEdit.allowed;  // reactive boolean

// @ts-expect-error — invalid entitlement
can('project:typo');
```

### 3.4 RLS policy output (Phase 2)

**File: `.vertz/generated/rls-policies.sql`**

```sql
-- Generated by @vertz/codegen — do not edit

-- Entitlement: project:delete
-- Rule: where({ createdBy: rules.user.id })
CREATE POLICY project_delete_owner ON projects FOR ALL
  USING (created_by = current_setting('app.user_id')::UUID);
```

## 4. Compiler: AccessAnalyzer

### 4.1 Detection

Follows the same pattern as `EntityAnalyzer`:

1. Find all `CallExpression` nodes in source files
2. Filter to calls where the identifier resolves to `@vertz/server` via `isFromImport()`
3. Match function name `defineAccess`
4. Extract the first argument as an object literal

### 4.2 Extraction

From the config object, extract:

| Source path | Extraction method | IR field |
|-------------|-------------------|----------|
| `entities` keys | `getProperties()` → `name` | `AccessIR.entities[].name` |
| `entities[name].roles` | `getArrayElements()` → `getStringValue()` | `AccessIR.entities[].roles` |
| `entitlements` keys | `getProperties()` → `name` | `AccessIR.entitlements[]` |
| `entitlements[name]` callback `r.where()` | See §4.3 | `AccessIR.whereClauses[]` |

**Diagnostics:**
- `ACCESS_NON_LITERAL_KEY` — entity/entitlement key is not a string literal (warning: skip, don't fail)
- `ACCESS_NON_LITERAL_ROLE` — role in array is not a string literal (warning)
- `ACCESS_DUPLICATE_ENTITLEMENT` — same entitlement key defined twice (error)
- `ACCESS_MULTIPLE_DEFINITIONS` — more than one `defineAccess()` call found (error)

### 4.3 Where-clause extraction (Phase 2)

For entitlements with callback form `(r) => ({ rules: [r.where({...})] })`:

1. Detect arrow function / function expression in entitlement value
2. Find `r.where()` calls within the function body
3. Extract the condition object literal:
   - Property name → column name
   - Property value:
     - `r.user.id` → `{ marker: 'user.id' }`
     - `r.user.tenantId` → `{ marker: 'user.tenantId' }`
     - String/number/boolean literal → `{ literal: value }`
     - Nested object → `{ nested: { ... } }` (recursive)
4. If extraction fails (complex expression, variable reference), emit `ACCESS_WHERE_NOT_TRANSLATABLE` warning

### 4.4 IR type

```ts
// Added to AppIR
interface AppIR {
  // ... existing fields
  access?: AccessIR;
}

interface AccessIR extends SourceLocation {
  entities: AccessEntityIR[];
  entitlements: string[];
  whereClauses: AccessWhereClauseIR[];  // Phase 2
}

interface AccessEntityIR {
  name: string;
  roles: string[];
}

interface AccessWhereClauseIR {
  entitlement: string;
  conditions: AccessWhereCondition[];
}

type AccessWhereCondition =
  | { kind: 'marker'; column: string; marker: 'user.id' | 'user.tenantId' }
  | { kind: 'literal'; column: string; value: string | number | boolean }
  | { kind: 'nested'; column: string; conditions: AccessWhereCondition[] };
```

## 5. Codegen: AccessTypesGenerator

### 5.1 Generator

New generator in `packages/codegen/src/generators/access-types-generator.ts`.

**Input:** `CodegenIR.access` (adapted from `AccessIR` via `ir-adapter.ts`)
**Output:** Single file `access.d.ts`

**Logic:**
1. If `ir.access` is undefined or has no entitlements → return `[]` (no file generated)
2. Build `Entitlement` union from `ir.access.entitlements`
3. Build `ResourceType` union from `ir.access.entities` names
4. Build `Role<T>` conditional type from `ir.access.entities` roles
5. Render module augmentations for `@vertz/server` and `@vertz/ui`

### 5.2 IR adapter extension

```ts
// In ir-adapter.ts adaptIR()
const access = appIR.access
  ? {
      entities: appIR.access.entities,
      entitlements: appIR.access.entitlements,
      whereClauses: appIR.access.whereClauses ?? [],
    }
  : undefined;

return {
  // ... existing fields
  access,
};
```

### 5.3 Registration

Add `AccessTypesGenerator` to the generator list in `packages/codegen/src/generate.ts`, after `RouterAugmentationGenerator`.

## 6. Codegen: RLS Policy Generator (Phase 2)

### 6.1 Generator

New generator in `packages/codegen/src/generators/rls-policy-generator.ts`.

**Input:** `CodegenIR.access.whereClauses`
**Output:** Single file `rls-policies.sql`

### 6.2 Translation rules

| Source condition | SQL output |
|------------------|------------|
| `{ createdBy: rules.user.id }` | `created_by = current_setting('app.user_id')::UUID` |
| `{ tenantId: rules.user.tenantId }` | `tenant_id = current_setting('app.tenant_id')::UUID` |
| `{ archived: false }` | `archived = false` |
| `{ status: 'active' }` | `status = 'active'` |
| `{ team: { status: 'active' } }` | `team_id IN (SELECT id FROM teams WHERE status = 'active')` |

### 6.3 Column name mapping

Column names in `rules.where()` use camelCase (JS convention). SQL uses snake_case. The generator converts:
- `createdBy` → `created_by`
- `tenantId` → `tenant_id`
- `userId` → `user_id`

Uses the same `toSnakeCase()` utility the framework already uses for table column mapping.

## 7. Type Flow Map

```
defineAccess() source code
    │
    ├─ .entities keys ──────────────► AccessIR.entities[].name
    │   └─ .roles array ───────────► AccessIR.entities[].roles
    │
    ├─ .entitlements keys ─────────► AccessIR.entitlements[]
    │   └─ callback r.where() ────► AccessIR.whereClauses[] (Phase 2)
    │
    ▼
AccessIR (in AppIR)
    │
    ▼ (ir-adapter.ts)
CodegenIR.access
    │
    ├──► AccessTypesGenerator
    │       │
    │       ├─ Entitlement union ──► ctx.can(entitlement) constraint
    │       │                        can(entitlement) constraint
    │       │
    │       ├─ ResourceType union ─► Type-level entity discrimination
    │       │
    │       └─ Role<T> conditional ► rules.role() constraint (future)
    │
    └──► RlsPolicyGenerator (Phase 2)
            │
            └─ CREATE POLICY SQL ──► Migration file for Postgres RLS
```

**Every generic flows to a consumer:**
- `Entitlement` → server `ctx.can()` + client `can()` parameter
- `ResourceType` → `Role<T>` type parameter
- `Role<T>` → `rules.role()` parameter (future augmentation)

## 8. Manifesto Alignment

### "If it builds, it works" (Principle #1)

This is the primary driver. Authorization entitlements are currently `string` — a gap in compile-time safety. After this work, a misspelled entitlement is a red squiggle, not a production bug.

### "One way to do things" (Principle #2)

There's exactly one way to define access: `defineAccess()`. The compiler reads it. The codegen types it. No alternative APIs, no manual `.d.ts` files.

### "AI agents are first-class users" (Principle #3)

Autocomplete on entitlement names means an LLM doesn't need to memorize or look up the valid entitlement strings. The type system guides it.

### What was rejected

- **Runtime type validation** — `defineAccess()` already validates at runtime. Adding compile-time types on top is additive, not replacing.
- **Decorator-based access** — Decorators break type flow. Function calls + string literals extracted by compiler is the Vertz way.
- **Manual `.d.ts` files** — Users should never hand-write access types. The compiler generates them from the single source of truth.

### Tradeoffs

- **Requires codegen step** — Types only appear after running `vertz generate`. Acceptable because all Vertz type generation works this way (entity types, router types).
- **String literals only** — The compiler can only extract literal strings, not computed or imported values. This is intentional: access definitions should be explicit and visible in one file.

## 9. Non-Goals

- **Runtime validation of `defineAccess()` config** — already implemented
- **Migration execution** — codegen generates SQL files, doesn't run them
- **Custom SQL in RLS policies** — only `rules.where()` declarative conditions
- **Hot type generation during dev server** — may require restart after changing `defineAccess()`; acceptable because access definitions change infrequently
- **`DenialReasonFor<E>` narrowed type** — deferred; requires tracking which denial layers (flag, plan, role, limit) apply per entitlement. Can be added later as a non-breaking enhancement.
- **Type-safe `rules.where()` column names** — requires cross-referencing entity models with entitlement rules. Deferred to a future enhancement (needs cross-file analysis or IR linking).

## 10. Unknowns

### 10.1 Module augmentation scope

**Question:** Should the augmentation override the existing `string` parameter on `can()`, or add an overload?

**Resolution plan:** Override via `declare module` augmentation. TypeScript merges interface declarations, so a more specific `entitlement: Entitlement` replaces `entitlement: string` when the `.d.ts` is included in the project's `tsconfig`. We need to verify this works correctly with the existing interface — POC in Phase 1.

### 10.2 Callback entitlements extraction

**Question:** Can the compiler extract entitlement keys when the config uses spread or variables?

```ts
const ents = { 'a:b': { roles: ['admin'] } };
defineAccess({ entitlements: { ...ents, 'c:d': { roles: ['user'] } } });
```

**Resolution:** Don't support this. Require literal object keys in the `defineAccess()` call. Emit `ACCESS_NON_LITERAL_KEY` warning for spread/computed keys. This is consistent with how the compiler handles other extractions (entity names, schema names).

### 10.3 Multiple `defineAccess()` calls

**Question:** What if the user calls `defineAccess()` in multiple files?

**Resolution:** Emit `ACCESS_MULTIPLE_DEFINITIONS` error. There should be exactly one access definition per application. This is the same constraint `AppAnalyzer` has for `vertz.app()`.

## 12. Implementation Phases

### Phase 1: AccessAnalyzer + AccessTypesGenerator

**Scope:** Compiler extracts `defineAccess()` → codegen generates typed `.d.ts`

**Acceptance criteria:**
1. `AccessAnalyzer` extracts entitlement keys, entity names, and roles from `defineAccess()` AST
2. `AccessIR` added to `AppIR`
3. `ir-adapter.ts` passes access data to `CodegenIR`
4. `AccessTypesGenerator` emits `access.d.ts` with `Entitlement`, `ResourceType`, `Role<T>`, and module augmentations
5. `@ts-expect-error` on invalid entitlement in both server `ctx.can()` and client `can()`
6. Compiler emits diagnostics for non-literal keys and duplicate definitions

**Vertical slice:** After this phase, a developer can run codegen and get autocomplete + type errors on entitlement names.

### Phase 2: RLS Policy Generation

**Scope:** Compiler extracts `rules.where()` conditions → codegen generates SQL

**Acceptance criteria:**
1. `AccessAnalyzer` extracts where-clause conditions from entitlement callbacks
2. `AccessIR.whereClauses` populated with structured condition data
3. `RlsPolicyGenerator` translates conditions to `CREATE POLICY` SQL
4. User markers (`rules.user.id`) resolve to `current_setting()` calls
5. Literal values generate equality conditions
6. Nested objects generate subqueries
7. Non-translatable rules emit warnings (no SQL generated)

**Vertical slice:** After this phase, RLS policies are auto-generated from the same `defineAccess()` config that drives runtime access checks.
