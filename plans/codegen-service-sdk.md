# Codegen: Service SDK Generation

## Problem

Services are a first-class server-side concept in Vertz — `service()` definitions with typed actions, route generation, access control — but the codegen pipeline ignores them. Developers must manually write HTTP calls to service endpoints instead of getting the same type-safe SDK they get for entities.

The `adaptIR` function hardcodes `modules: []`, there's no `ServiceSdkGenerator`, and the `ClientGenerator` only exposes entity SDKs. This gap breaks the "schema → database → API → client → UI" promise from the Vision.

## API Surface

### Developer-defined service (input)

```typescript
// api/services/notifications.service.ts
import { s } from '@vertz/schema';
import { action, service } from '@vertz/server';
import { rules } from '@vertz/auth/rules';

const sendBody = s.object({
  to: s.string(),
  subject: s.string(),
  body: s.string(),
});

const sendResponse = s.object({
  messageId: s.string(),
  status: s.enum(['queued', 'sent', 'failed']),
});

const statusResponse = s.object({
  messageId: s.string(),
  status: s.enum(['queued', 'sent', 'failed', 'delivered', 'bounced']),
  updatedAt: s.string(),
});

export const notifications = service('notifications', {
  access: {
    send: rules.entitlement('notification:send'),
    status: rules.authenticated(),
  },
  actions: {
    send: action({
      body: sendBody,
      response: sendResponse,
      handler: async (input, ctx) => {
        // ...
        return { messageId: '123', status: 'queued' };
      },
    }),
    status: action({
      method: 'GET',
      path: 'notifications/status/:messageId',
      response: statusResponse,
      handler: async (_input, ctx) => {
        const { messageId } = ctx.request.params;
        // ...
        return { messageId, status: 'sent', updatedAt: new Date().toISOString() };
      },
    }),
  },
});
```

### Generated SDK (output)

All service actions use `createDescriptor` — services don't participate in the entity store, so `createMutationDescriptor` (which powers optimistic updates and cache invalidation) is not used. This avoids extending `MutationMeta` with a `serviceType` field and keeps the boundary clean: entities own the mutation pipeline, services are plain RPC.

```typescript
// #generated/services/notifications.ts
// Generated by @vertz/codegen — do not edit

import type { SendNotificationsInput, SendNotificationsOutput, StatusNotificationsOutput } from '../types/notifications';
import { type FetchClient, createDescriptor } from '@vertz/fetch';

export function createNotificationsSdk(client: FetchClient) {
  return {
    send: Object.assign(
      (body: SendNotificationsInput) =>
        createDescriptor(
          'POST',
          '/notifications/send',
          () => client.post<SendNotificationsOutput>('/notifications/send', body),
        ),
      { url: '/notifications/send', method: 'POST' as const },
    ),
    status: Object.assign(
      (messageId: string) =>
        createDescriptor(
          'GET',
          `/notifications/status/${messageId}`,
          () => client.get<StatusNotificationsOutput>(`/notifications/status/${messageId}`),
        ),
      { url: '/notifications/status/:messageId', method: 'GET' as const },
    ),
  };
}

export type NotificationsSdk = ReturnType<typeof createNotificationsSdk>;
```

### Usage

```typescript
import { createClient } from '#generated';

const api = createClient();

// Type-safe, identical DX to entities
const result = await api.notifications.send({
  to: 'user@example.com',
  subject: 'Welcome',
  body: 'Hello!',
});

// GET action with path params
const status = await api.notifications.status('msg-123');

// Works with query() in the UI
const msgStatus = query(() => api.notifications.status(messageId));
```

## Manifesto Alignment

### If it builds, it works (Principle 1)
Service SDK types flow from the `s.object()` schemas through the compiler IR into generated types. If the SDK call compiles, the server will accept it.

### One way to do things (Principle 2)
Entities and services follow the same SDK pattern — `Object.assign` + descriptors + `FetchClient`. No separate calling convention for services.

### AI agents are first-class users (Principle 3)
An LLM sees `api.notifications.send(...)` with full autocomplete. Identical to `api.todos.create(...)`. No ambiguity about how to call a service.

### If it builds, it works — again (Type Flow)
Types flow: `s.object()` → compiler `SchemaRef` → codegen `CodegenServiceAction` → generated TypeScript interface → SDK method parameter type. One chain, compiler-verified.

## Non-Goals

- **Service-to-service SDK** — This is a client SDK. Server-side cross-service calls use `ctx.entities` via inject, not the HTTP SDK.
- **Streaming/SSE support** — Services that return streams are out of scope for this design. The codegen pipeline doesn't handle streaming for entities either.
- **Optimistic updates / MutationDescriptor for services** — Entity mutations use `createMutationDescriptor` which powers optimistic updates and cache invalidation via the entity store. Services are arbitrary RPC with no entity store — all service actions use `createDescriptor` regardless of HTTP method. This avoids extending `MutationMeta` with `serviceType`.
- **Query schema on service actions** — The `ServiceActionDef` type supports `body` and `response` schemas but not `query`. Typed query params for service GET actions are out of scope. Developers who need query params can use the raw `FetchClient` or add `query` to the service action API in a future iteration.
- **Service schemas in entity-schema-manifest.json** — The entity schema manifest is entity-specific (powers entity store, optimistic updates). Services don't participate in this.
- **Custom service middleware** — Service middleware is a server-side concern, not relevant to SDK generation.

## Unknowns

### 1. ServiceIR is too shallow for SDK generation

**Status:** Needs resolution in Phase 1.

The compiler's `ServiceIR` currently captures only `methods: ServiceMethodIR[]` with name/params/returnType — designed for an older `methods()` factory pattern. The current `service()` API uses `actions:` with `action()` helpers that have `body`, `response`, `method`, `path`.

**Resolution approach:** Enhance `ServiceIR` to carry action-level detail:

```typescript
// Proposed enhancement to compiler IR
export interface ServiceActionIR {
  name: string;
  method: HttpMethod;
  path?: string;
  body?: SchemaRef;
  response?: SchemaRef;
}

export interface ServiceIR extends SourceLocation {
  name: string;
  moduleName: string;
  inject: InjectRef[];
  actions: ServiceActionIR[];  // replaces methods: ServiceMethodIR[]
  access: Record<string, EntityAccessRuleKind>;
}
```

The `ServiceAnalyzer` must be updated to parse the `actions:` config object and extract `action()` call properties — following the same pattern as entity action extraction.

### 2. ServiceAnalyzer architecture mismatch

**Status:** Needs resolution in Phase 1.

The current `ServiceAnalyzer.analyzeForModule()` uses `findMethodCallsOnVariable(file, moduleDefVarName, 'service')` — it looks for `moduleDef.service(...)` calls. But the runtime API is standalone: `service('name', { ... })` imported from `@vertz/server`. The analyzer also extracts the service name from the **variable name** (`getVariableNameForCall`), but the canonical name comes from the first string argument to `service()`.

**Resolution approach:** Restructure the analyzer to find standalone `service()` calls imported from `@vertz/server`, similar to how the entity analyzer finds `entity()` calls. Extract the service name from the first string argument. The `moduleName` field on `ServiceIR` can be dropped or made optional since services are standalone.

### 3. Services never populate in AppIR

**Status:** Needs resolution in Phase 1.

The `ServiceAnalyzer` exists but is not wired into `Compiler.analyze()`. The `AppIR.modules[].services` array is always empty in practice. Phase 1 must wire the analyzer into the compiler pipeline.

**Resolution approach:** Two options:
- **(a)** Add `services: ServiceIR[]` as a top-level field on `AppIR` (paralleling `entities: EntityIR[]`), wire the standalone `ServiceAnalyzer` into `Compiler.analyze()`.
- **(b)** Keep services inside `ModuleIR` and wire the module-level analysis to populate them.

Option (a) is recommended — it matches how entities work (top-level in AppIR, flat list) and simplifies the `adaptIR` function in codegen. If services are top-level, the adapter iterates `appIR.services` directly instead of flattening from modules.

### 4. Path params in service actions

**Status:** Resolved by design.

Service actions can have path params (e.g., `notifications/status/:messageId` or `users/:userId/messages/:messageId`). The SDK generates function signatures with these params extracted.

The SDK generator will:
- Parse `:paramName` segments from the action path using regex `/:([a-zA-Z][a-zA-Z0-9]*)/g`
- Generate typed `string` function parameters for each, ordered by appearance in the path
- Build template literal paths: `` `/notifications/status/${messageId}` ``
- For multiple params: `(userId: string, messageId: string) => createDescriptor('GET', \`/users/${userId}/messages/${messageId}\`, ...)`

### 5. Access rule treatment for SDK inclusion

**Status:** Resolved by design.

The codegen includes a service action in the SDK if it has **any** access rule defined — whether it's a `rules.*` descriptor or a function (`() => true`). The codegen only looks at the `EntityAccessRuleKind` discriminant from the compiler IR: `'none'` (no rule → skip, deny-by-default), `'false'` (explicitly disabled → skip), or `'function'` (has a rule → include). This matches the route-generator behavior: actions without rules get no route, actions with `false` get a 405, actions with any other rule get a handler.

### 6. Path prefix conventions

**Status:** Resolved by design.

- **Default path:** `/{serviceName}/{actionName}` — no leading slash in the definition, the SDK prepends the base URL via `FetchClient.baseURL`.
- **Custom path:** The `path` property on `action()` replaces the default entirely. It's relative to the API prefix (e.g., `path: 'notifications/status/:messageId'` → route is `{apiPrefix}/notifications/status/:messageId`).
- **In the SDK:** All paths are relative to `FetchClient.baseURL` (which defaults to `/api`). The SDK stores the path without the `/api` prefix.

## Implementation Plan

### Phase 1: Compiler IR Enhancement + Wiring

Enhance `ServiceIR`, restructure `ServiceAnalyzer` to find standalone `service()` calls, and wire it into the compiler pipeline so services actually appear in `AppIR`.

**Task 1a: IR types + analyzer restructure**

**Files:**
- `packages/compiler/src/ir/types.ts` — Add `ServiceActionIR`, update `ServiceIR`, add `services: ServiceIR[]` to `AppIR`
- `packages/compiler/src/analyzers/service-analyzer.ts` — Restructure to find standalone `service()` from `@vertz/server`
- `packages/compiler/src/analyzers/__tests__/service-analyzer.test.ts` — Tests

**Key changes:**
- Add `ServiceActionIR` with `name`, `method`, `path?`, `body?`, `response?` (all `SchemaRef`)
- Add `access: Record<string, EntityAccessRuleKind>` to `ServiceIR`
- Replace `methods: ServiceMethodIR[]` with `actions: ServiceActionIR[]`
- Add `services: ServiceIR[]` to `AppIR` (top-level, paralleling `entities`)
- Restructure `ServiceAnalyzer` to scan for standalone `service('name', {...})` calls imported from `@vertz/server` — model after `EntityAnalyzer.findEntityCalls()` pattern
- Extract service name from the first string argument (not variable name)
- Parse the `actions:` property, extract `action()` call properties (method, path, body, response)
- Parse `access:` property to extract per-action access rule kinds

**Task 1b: Wire into compiler pipeline**

**Files:**
- `packages/compiler/src/compiler.ts` — Add ServiceAnalyzer to `analyze()` pipeline
- `packages/compiler/src/index.ts` — Export new types if needed

**Key changes:**
- Instantiate `ServiceAnalyzer` in `Compiler.analyze()` alongside entity/schema/auth analyzers
- Populate `appIR.services` from analyzer results
- Export `ServiceActionIR` from `@vertz/compiler` for codegen consumption

### Phase 2: CodegenIR Types + IR Adapter

Add service types to CodegenIR and wire `adaptIR` to process services from `AppIR.modules`.

**Files:**
- `packages/codegen/src/types.ts` — Add `CodegenServiceModule`, `CodegenServiceAction`
- `packages/codegen/src/ir-adapter.ts` — Process `appIR.modules[].services` into `CodegenIR.services`
- `packages/codegen/src/__tests__/ir-adapter.test.ts` — Tests

**Key changes:**
```typescript
export interface CodegenServiceModule {
  serviceName: string;
  actions: CodegenServiceAction[];
}

export interface CodegenServiceAction {
  name: string;
  method: string;
  path: string;
  inputSchema?: string;
  outputSchema?: string;
  pathParams?: string[];
  resolvedInputFields?: CodegenResolvedField[];
  resolvedOutputFields?: CodegenResolvedField[];
}
```

- Add `services?: CodegenServiceModule[]` to `CodegenIR` (optional to avoid breaking existing test fixtures)
- In `adaptIR`, iterate `appIR.services` (top-level, added in Phase 1), map to `CodegenServiceModule`
- Filter actions by access rules: skip `'none'` (deny-by-default) and `'false'` (explicitly disabled), include `'function'` (has a rule)
- Resolve schema names following entity naming convention: `${ActionPascal}${ServicePascal}Input/Output`
- Extract `pathParams` from action paths using `:paramName` regex

### Phase 3: Service SDK Generator

Generate service SDK files (parallel to `EntitySdkGenerator`).

**Files:**
- `packages/codegen/src/generators/service-sdk-generator.ts` — New generator
- `packages/codegen/src/generators/service-types-generator.ts` — New generator (input/output types)
- `packages/codegen/src/__tests__/generators.test.ts` — Add service tests

**Key changes:**
- `ServiceSdkGenerator` implements `Generator`, produces `services/{serviceName}.ts` + `services/index.ts`
- Each action maps to a method on the returned object:
  - All actions use `createDescriptor` (no `createMutationDescriptor` — services don't participate in entity store)
  - GET/DELETE without body: `(pathParams...) => createDescriptor(method, path, () => client.get/delete(...))`
  - POST/PUT/PATCH with body: `(pathParams..., body: InputType) => createDescriptor(method, path, () => client.post/put/patch(..., body), body)`
  - Path params come before body in function signature, ordered by appearance in path
  - `Object.assign` pattern with `{ url, method }` metadata
- `ServiceTypesGenerator` produces `types/{serviceName}.ts` for input/output interfaces from resolved fields
- GET actions with only path params generate no Input type (only Output type)

### Phase 4: Client Integration + Orchestrator

Wire service SDKs into the client generator and orchestrator.

**Files:**
- `packages/codegen/src/generators/client-generator.ts` — Import + expose service SDKs
- `packages/codegen/src/generate.ts` — Add service generators to pipeline
- `packages/codegen/src/__tests__/generate.test.ts` — Integration tests

**Key changes:**
- `ClientGenerator` imports `create${Pascal}Sdk` from `./services/${name}`, adds to `createClient()` return object
- Service SDKs don't receive the `optimistic` handler — `createNotificationsSdk(client)` vs entity `createTodosSdk(client, optimistic)`
- `runTypescriptGenerator` runs `ServiceTypesGenerator` and `ServiceSdkGenerator` alongside entity generators
- README generator lists service methods alongside entity methods
- No new subpath imports (`#generated/services`) — services are accessed via `createClient()`, types via `#generated/types` (same as entities, no asymmetry)
