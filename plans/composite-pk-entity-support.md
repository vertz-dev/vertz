# Composite Primary Key Support in Entity CRUD

**Issue:** [#1776](https://github.com/vertz-dev/vertz/issues/1776)
**Status:** Draft (Rev 2 — addressing DX, Product, and Technical review feedback)
**Date:** 2026-04-04
**Depends on:** [#1760](https://github.com/vertz-dev/vertz/issues/1760) (composite PK schema — merged)

## Problem

`createCrudHandlers()` and `resolvePrimaryKey()` in `tenant-chain.ts` throw when an entity has a composite primary key. This blocks natural modeling of join tables (e.g., `tenant_members(tenantId, userId)`) and other composite-PK entities.

The schema layer already supports composite PKs (#1760). The TYPE system correctly includes composite PK columns in `$create_input` (via `AutoGeneratedPKKeys` — only `primary + hasDefault` columns are excluded). However, the RUNTIME validation schemas from `tableToSchemas()` unconditionally exclude ALL `meta.primary` columns from `apiCreateBody`, creating a type/runtime mismatch.

This issue lifts the entity layer restriction so composite-PK tables can participate in CRUD operations, route generation, and tenant chain resolution, and fixes the schema derivation bug.

## API Surface

```ts
const projectMembers = d.table('project_members', {
  projectId: d.uuid(),

```

### Generated routes (new behavior)

Composite-PK entities generate multi-segment URL paths using PK column order from the `primaryKey` array:

```
# Single PK (unchanged)
GET    /tasks/:id
PATCH  /tasks/:id
DELETE /tasks/:id

# Composite PK — segments in primaryKey array order
GET    /project-members/:projectId/:userId
PATCH  /project-members/:projectId/:userId
DELETE /project-members/:projectId/:userId

# Custom actions on composite-PK entities
POST   /project-members/:projectId/:userId/deactivate

# List + Create remain unchanged (no ID in path)
GET    /project-members
POST   /project-members
```

At startup, the framework logs composite-PK routes for discoverability:

```
[vertz] Entity "project-member" routes: /api/project-members/:projectId/:userId (composite PK)
```

### CRUD handler interface (widened ID type)

```ts
// New type alias
type EntityId = string | Record<string, string>;

export interface CrudHandlers<TModel extends ModelDef = ModelDef> {
  list(ctx: EntityContext<TModel>, options?: ListOptions):
    Promise<Result<CrudResult<ListResult<TModel['table']['$response']>>, EntityError>>;

  // id: string → EntityId
  get(ctx: EntityContext<TModel>, id: EntityId, options?: GetOptions):
    Promise<Result<CrudResult<TModel['table']['$response']>, EntityError>>;

  create(ctx: EntityContext<TModel>, data: Record<string, unknown>):
    Promise<Result<CrudResult<TModel['table']['$response']>, EntityError>>;

  // id: string → EntityId
  update(ctx: EntityContext<TModel>, id: EntityId, data: Record<string, unknown>):
    Promise<Result<CrudResult<TModel['table']['$response']>, EntityError>>;

  // id: string → EntityId
  delete(ctx: EntityContext<TModel>, id: EntityId):
    Promise<Result<CrudResult<null>, EntityError>>;
}
```

**Known type-safety gap:** `EntityId = string | Record<string, string>` is a loose union — it doesn't discriminate by table. Passing a `Record` to a single-PK entity compiles but fails at runtime. A typed `EntityIdFor<TModel>` that derives the ID shape from `TModel['table']['_primaryKey']` is a follow-up enhancement. The runtime validates composite key completeness, so mismatches surface as clear error messages, not silent failures. This is acceptable for v1 given pre-v1 status (no external consumers), but should be tightened before v1.

### DB adapter interface (widened ID type)

```ts
export interface EntityDbAdapter<TEntry extends ModelEntry = ModelEntry> {
  get(id: string | Record<string, string>, options?: GetOptions<TEntry>):
    Promise<TEntry['table']['$response'] | null>;

  update(
    id: string | Record<string, string>,
    data: TEntry['table']['$update_input'],
    options?: UpdateOptions<TEntry>,
  ): Promise<TEntry['table']['$response']>;

  delete(
    id: string | Record<string, string>,
    options?: DeleteOptions<TEntry>,
  ): Promise<TEntry['table']['$response'] | null>;
}
```

When `id` is a `Record`, the bridge adapter spreads it into the WHERE clause:

```ts
// Single PK (backward compatible)
db.get('uuid-123')          // → WHERE "id" = 'uuid-123'

// Composite PK
db.get({ projectId: 'p1', userId: 'u1' })
  // → WHERE "projectId" = 'p1' AND "userId" = 'u1'
```

### EntityOperations interface (widened ID type)

The programmatic entity API used by hooks and cross-entity injection (`ctx.entities.xxx.get(id)`) must also accept composite IDs:

```ts
export interface EntityOperations {
  get(id: EntityId, options?: GetOptions): Promise<...>;
  update(id: EntityId, data: Record<string, unknown>): Promise<...>;
  delete(id: EntityId): Promise<...>;
}
```

### Cursor pagination (composite encoding)

```ts
// Single PK cursor (unchanged)
nextCursor: "uuid-123"

// Composite PK cursor — JSON-encoded object with PK columns
nextCursor: '{"projectId":"p1","userId":"u1"}'
```

The cursor is a JSON-encoded object with PK column names as keys. This is:
- Readable in logs and debuggable (developer sees named keys, not opaque tuples)
- What a developer or LLM would guess on the first try
- Deterministic (keys serialized in `_primaryKey` array order)
- Distinguishable from single-PK cursors (starts with `{`, not a UUID/number)
- Backward compatible (single PK cursors are still plain strings)

**Format detection:** Check if the string starts with `{`, parse as JSON, validate that the result is an object with string values and keys matching the entity's PK columns. UUIDs and numeric IDs never start with `{`, so this is unambiguous. Structural validation prevents false positives from text PKs that happen to start with `{`.

**Invalid cursor handling:** If a cursor fails format detection or structural validation for the entity, return a 400 error with a descriptive message.

### Error messages

```ts
// Composite PK entity called with single string ID
crudHandlers.get(ctx, 'single-string');
// Error: Entity "project-member" requires composite key { projectId, userId }.
//        Got a single string ID. Use: handlers.get(ctx, { projectId: "...", userId: "..." })

// Missing PK column in composite ID
crudHandlers.get(ctx, { projectId: 'p1' });
// Error: Entity "project-member" requires composite key { projectId, userId }.
//        Missing key column: "userId".

// Composite PK entity not found
crudHandlers.get(ctx, { projectId: 'p1', userId: 'u1' });
// Error: project-member with key { projectId: "p1", userId: "u1" } not found
```

## Manifesto Alignment

### "If it builds, it works" (Principle 1)

The composite PK column names are type-constrained at the schema level (`primaryKey: ['projectId', 'userId']` — only valid column names accepted). Route generation derives path segments from the same typed array. The compiler catches invalid PK column references at build time. The `EntityId` union is a known type-safety gap at the handler level, documented with a follow-up plan to derive ID shape from the model.

### "One way to do things" (Principle 2)

There is exactly one way to define composite PKs (`primaryKey` option on `d.table()`) and exactly one way to address composite-PK entities in URLs (path segments in PK array order). No query-param alternative, no mixed encoding.

### "AI agents are first-class users" (Principle 3)

An LLM sees `GET /project-members/:projectId/:userId` and knows exactly what params to pass. The PK column names ARE the path param names — zero ambiguity. Compared to encoded alternatives (`/project-members/p1~u1`), the explicit path is what an LLM would guess on the first try.

### "Performance is not optional" (Principle 7)

Composite PK lookups use `WHERE pk1 = ? AND pk2 = ?` — direct index usage, no additional queries. The chain resolution (for indirectly-scoped composite entities) follows the same BFS algorithm with no extra hops.

### Rejected alternatives

| Alternative | Why rejected |
|---|---|
| Encoded single-segment (`/project-members/p1~u1`) | Ambiguous delimiter; LLMs wouldn't guess the encoding; not self-describing |
| Query params (`?projectId=p1&userId=u1`) | Not RESTful for single-resource operations; mixes filtering with identity |
| Surrogate PK requirement | Forces unnatural schema design; the schema layer already supports composite PKs |
| `TenantChainHop.targetColumns: string[]` | Requires composite FKs (non-goal); single-column FKs can't reference composite PKs correctly (see Deferred Acceptance Criteria) |
| Tuple cursor format (`[["col","val"]]`) | Less readable in logs; developers guess object format first; harder to construct manually for testing |

## Non-Goals

1. **Composite foreign keys** — `ref.one()` creates single-column FKs. Multi-column FKs are a separate design.
2. **Composite-PK tables as intermediate chain hops** — A table with composite PK cannot be an intermediate hop in another entity's tenant chain, because `ref.one` FKs can only reference single-column PKs. The `resolvePrimaryKey` function will continue to throw for composite-PK targets encountered during chain BFS. This is correct behavior: you can't have a single FK column reference a composite PK. (See Deferred Acceptance Criteria.)
3. **Changing `TenantChainHop.targetColumn` to `targetColumns: string[]`** — Not needed given non-goal #2. All chain hops target single-PK tables. The composite PK is only on the entity itself (the chain's starting point), whose PK is never resolved during chain traversal. (See Deferred Acceptance Criteria.)
4. **Bridge adapter PK column name assumption** — The bridge adapter currently hardcodes `{ id }` for single-PK lookups. Fixing this for tables where the single PK isn't named `id` is a separate issue. Composite PK lookups use actual column names and are unaffected.
5. **Typed `EntityIdFor<TModel>` generic** — Deriving the ID shape from the model's `_primaryKey` at the type level is a follow-up enhancement. For v1, `EntityId = string | Record<string, string>` with runtime validation is acceptable.
6. **Client SDK / `@vertz/fetch` composite paths** — The client SDK will need to build composite URL paths (`/${projectId}/${userId}`). This is a follow-up once server-side routes land.
7. **OpenAPI spec generation for composite paths** — OpenAPI will skip composite-PK entities with a warning until composite path support is added. Follow-up issue.

## Deferred Acceptance Criteria

The issue (#1776) includes acceptance criteria that require composite foreign keys:
- "`TenantChainHop` supports multiple target columns"
- "JOIN conditions use all PK columns (not just the first)"
- "Integration test: composite-PK table in a tenant chain resolves correctly" (as an intermediate hop)

These are blocked by the single-column FK model (`ref.one()` creates one FK column; a single FK cannot reference a composite PK). They are deferred to a future "composite FK" design. The issue should be updated to reflect this narrower scope.

**What this design DOES deliver for tenant chains:** A composite-PK entity as the chain ORIGIN works correctly — the chain follows the entity's outgoing `ref.one` FKs to single-PK target tables. The entity's own composite PK is never resolved during chain BFS.

## Unknowns

### 1. Cursor decode ambiguity — **Resolved: format detection with structural validation**

**Question:** How does the cursor decoder distinguish between a single-PK cursor (`"uuid-123"`) and a composite cursor (`'{"projectId":"p1"}'`)?

**Resolution:** Check if the string starts with `{` and is valid JSON. Parse and validate: must be an object with string values and keys matching the entity's `_primaryKey` columns. If parsing or validation fails, return a 400 error. This is unambiguous because UUIDs and numeric IDs never start with `{`.

### 2. Composite PK validation at route layer — **Resolved: validate at CRUD handler**

**Question:** Should path params be validated in the route handler or the CRUD handler?

**Resolution:** The CRUD handler already validates input. Add PK column completeness checking there — the route layer only extracts params. This keeps the route generator thin and the CRUD handler as the single validation point.

## Type Flow Map

```
d.table('project_members', cols, { primaryKey: ['projectId', 'userId'] })
  │
  ├── TableDef._primaryKey: readonly ['projectId', 'userId']
  │     │
  │     ├── tableToSchemas() reads _primaryKey or scans _meta
  │     │     │
  │     │     ├── apiCreateBody: INCLUDES projectId & userId (primary=true, hasDefault=false)
  │     │     │     (Bug fix: current code excludes ALL primary columns)
  │     │     │
  │     │     └── apiUpdateBody: EXCLUDES projectId & userId (PK columns never updatable)
  │     │
  │     ├── createCrudHandlers() reads _primaryKey → determines composite vs single
  │     │     │
  │     │     ├── Route generator: builds '/:projectId/:userId' path segments
  │     │     │     │
  │     │     │     └── Handler: getParams(ctx) → { projectId: string, userId: string }
  │     │     │           │
  │     │     │           └── Constructs Record<string, string> → passes to CRUD handler
  │     │     │
  │     │     ├── CRUD handler.get(ctx, { projectId, userId })
  │     │     │     │
  │     │     │     └── Bridge adapter.get({ projectId: 'p1', userId: 'u1' })
  │     │     │           │
  │     │     │           └── delegate.get({ where: { projectId: 'p1', userId: 'u1' } })
  │     │     │                 │
  │     │     │                 └── SQL: WHERE "projectId" = $1 AND "userId" = $2
  │     │     │
  │     │     ├── Cursor: reads _primaryKey → ['projectId', 'userId']
  │     │     │     │
  │     │     │     └── nextCursor = JSON.stringify({ projectId: lastRow.projectId, ... })
  │     │     │
  │     │     └── notFound(id): formats as "key { projectId: 'p1', userId: 'u1' }"
  │     │
  │     └── resolveTenantChain(): entity's own PK NOT used in chain BFS
  │           (only target PKs are resolved; entity is the starting point)
  │
  └── ColumnMetadata.primary: true  (on both projectId and userId)
        ColumnMetadata.hasDefault: false  (composite PK — user-supplied)
```
