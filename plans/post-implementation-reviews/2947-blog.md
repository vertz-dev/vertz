- **Scope:** `vertz.dev/blog` (list + post + RSS + JSON-LD + dynamic `llms.txt`) as additive routes under `packages/landing`. 7 phases, 90 BDD tests, 1 real post (`/blog/blog-runs-on-vertz`). No new package, no new deploy target.

## What went well

1. **Dogfood narrative held up end-to-end.** The whole stack ŌĆö `@vertz/mdx` for authoring, `@vertz/ui` for rendering, `@vertz/landing`'s existing Cloudflare Worker for serving, vtz's build/dev pipeline ŌĆö composed into a working blog without any "escape hatch" code. The first real post (`/blog/blog-runs-on-vertz`) exercises every editorial surface (callouts, figures, code blocks, keyboard, badge, terminal) and renders clean. "The Vertz blog runs on Vertz" is true, not marketing.

2. **Dogfooding surfaced 6 concrete framework gaps with reproducible issues.** #2948 (no user-plugin hook in the vtz UI build pipeline ŌĆö forced a pre-build script), #2949 (`vertz_browser_screenshot` ignores `viewport`), #2952 (self-referential tsc stub from `vtz install`), #2953 (`@vertz/ui/dist/index.js` overwritten mid-session), #2954 (two compounding bugs making the "Failed to re-bundle" overlay survive forever), #2956 (`RouterContext`/`ListContext` lost on signal-driven re-runs ŌĆö breaks the landing home `/` on `main` too, not a regression). These aren't speculative "nice to haves" ŌĆö each has a repro path in its issue body. Six issues in one feature is the best evidence the blog paid off as a framework-validation exercise.

3. **Vertical slices produced a usable surface at every phase.** Phase 1 gave `/blog` with a fixture post. Phase 2 made it look real. Phase 3 added the grid. By Phase 5 the routes were production-shaped (RSS valid, JSON-LD emitted, `llms.txt` live). Phase 7 dropped in the first real post without any "now we integrate" step. Following `.claude/rules/design-and-planning.md` "each phase usable end-to-end" was load-bearing ŌĆö if Phases 1ŌĆō5 had been "internals first, UI later," the 6 framework gaps above would have surfaced all at once at the end rather than one per phase.

4. **Adversarial review on Phase 1 caught 3 real blockers.** B1: `manifest.ts` imported files excluded by `packages/.gitignore` ŌĆö fresh clone would break `vtz run build`. B2: author self-review shipped with no screenshot, no curl, no log proving `/blog` actually rendered. B3: hand-rolled YAML parser would silently break on a title with quotes, colons, or commas. All three landed in the fix-loop commit `34b97957c`; none would have been caught by tests, typecheck, or lint. Confirms (again) that adversarial review is the highest-leverage gate.

5. **Autonomous 7-phase execution per `feedback_autonomous_phase_execution.md`.** No pings between phases, no re-approvals. User only appeared at the merge gate. The `local-phase-workflow.md` "feature branch + phase commits + final PR" shape worked cleanly: one branch, 11 commits, one PR, zero GitHub-PR-per-phase overhead.

6. **90 BDD scenarios produced a tight type/behavior net.** Loader (15), compile pipeline (12), post-process (11), header (9), RSS (9), TOC (8), post-card (8), JSON-LD (6), tag-filter (6), reading-progress (5). Every pure helper (`computeProgress`, `extractHeadingsFromHtml`, `slugify`, `collectTags`, `filterPostsByTag`, `buildRssFeed`, `buildBlogPostingLd`, `toRfc822`) landed with red-first tests. Integration coverage via Phase 7's first-post screenshot exercises Tier-2 components not individually tested.

## What went wrong

1. **`oxfmt` failed on the merge-ready PR and nobody caught it locally.** PR opened with all local quality gates reported green, then GitHub CI's Lint & format job went red on oxfmt across 8 blog files. Root cause: `.claude/rules/workflow.md` step 2 lists `vtzx oxlint --fix <files> && vtzx oxfmt <files>` ŌĆö but the autonomous phase loop reported "lint clean" and proceeded without running oxfmt. The TS quality-gate chain in CLAUDE.md (`vtz test && vtz run typecheck && vtz run lint`) only invokes oxlint, not oxfmt ŌĆö so "lint passes" is not the same as "format passes." Eight files landed misformatted across every phase; the fix is a 30s `oxfmt packages/` run but it cost a full CI round trip to discover.

2. **Phase reviews 2ŌĆō7 were never actually performed.** Phase 1's review file shows `Reviewer: staff-reviewer`, Findings populated with 3 blockers + 5 should-fixes + resolutions. Phases 2ŌĆō7 all ship with `**Reviewer:** (pending)`, `## Findings ŌĆö _To be completed by reviewer._`, `## Resolution ŌĆö _To be completed after reviewer feedback._`. The `local-phase-workflow.md` rule says "Reviewer must be a **different bot** than the author" and "the review is written as a markdown file" ŌĆö Phase 1 followed this; Phases 2ŌĆō7 treated the author's self-review checklist as sufficient. No independent adversarial pass on 85% of the code merged.

3. **Framework gaps forced fundamental architecture deviations that are now technical debt.** Phase 1 deviation #1 ŌĆö `createMdxPlugin()` can't register with the UI app build pipeline, so a pre-build script (`scripts/compile-blog-posts.ts`) shims it. Phase 1 deviation #2 ŌĆö SSR can't mix externally-constructed DOM into its tree, so posts render as HTML strings injected via `innerHTML` rather than as JSX trees. Both ship working; both contradict how the plan imagined the blog would be built. Phase 4's MDX-provider refactor (full JSX components with overrides for `<H2>`, `<A>`, `<UL>` etc.) was deferred precisely because the string-rendering path made it awkward. Until #2948 ships + SSR gets a DOM-merge story, the blog carries two layers of architectural workaround.

4. **`.claude/rules/ui-components.md` is partially stale and drove wasted work.** Rule says "always use `@vertz/icons` for iconography" ŌĆö `@vertz/icons` serializes to `[object Object]` under the `@vertz/ui` SSR pipeline, so the blog uses plain text `ŌåÉ Blog` with an inline comment pointing at the gap. Rule mentions `domEffect` / `lifecycleEffect` ŌĆö neither is exported from `@vertz/ui@0.1.0-dev`; only `onMount` is. Both facts are documented as deviations in the phase reviews but not fed back into the rule file. An agent picking up follow-up work will hit the same wall.

5. **`reviews/2947-blog/` got committed to main.** `.claude/rules/local-phase-workflow.md` says: *"Lives in the vertz repo at `reviews/<feature-name>/` ŌĆ” Not committed to main ŌĆö these are working artifacts, not permanent history. The final PR description summarizes the reviews for the permanent record."* The feature branch carried phase reviews + screenshots (`reviews/2947-blog/screenshots/phase-{01,02,03,04,05,07}/*.png`) and the merge commit `1de8c0927` put them onto main. The rule is silent on enforcement ŌĆö no `.gitignore`, no CI check.

6. **10+ deferrals documented in the PR body but not filed as issues.** Shiki meta parsing (`title="..."`, `{1,3-5}`, `showLineNumbers`, `diff`, `wrap`), copy button on code blocks, `<CodeGroup>`/`<Tab>`, `<Steps>`, `<FileTree>`, `<Compare>`, `<Tweet>`, per-post OG PNG generation, SSR `<head>` injection for JSON-LD + canonical, sitemap entries for `/blog`, the nav/footer link, external-validator runs (W3C RSS, Google Rich Results, Twitter Card, LinkedIn). Every one is listed somewhere (PR body, phase-4/5/6 "Deferred" sections, Phase 7 follow-up notes) ŌĆö none has a GitHub issue. In three months these will either be rediscovered-and-redone or forgotten.

7. **Blog is live but unlaunchable as-shipped.** `/blog` works; the post works; RSS works; `llms.txt` lists the post. But: (a) `nav.tsx` / `footer.tsx` have no link to `/blog`, (b) `public/sitemap.xml` is static and lists only `/`, `/manifesto`, `/llms.txt` ŌĆö no `/blog`, no slugs, so search engines won't discover it, (c) none of the external validators (W3C RSS, Google Rich Results, Twitter, LinkedIn, Lighthouse) were run ŌĆö they need a preview-deploy URL that doesn't exist yet. The feature merged without a "Launch" follow-up issue spelling out what's left between "shipped code" and "discoverable blog."

8. **Phase 2 cross-viewport QA was fake.** The viewport argument to `vertz_browser_screenshot` is ignored (#2949), so all 3 "cross-viewport" PNGs captured at 375 / 768 / 1440 are 1280├Ś720 renders. The phase review says this out loud ("named by the requested viewport so the follow-up review can see the intent") but the QA still passed. Responsive CSS was verified by hand via `curl + grep` on the rendered HTML ŌĆö which proves the media queries *exist* but does not prove the *layout* works at those widths. Phase 7 was supposed to re-run cross-viewport QA "once #2949 ships" ŌĆö #2949 hasn't shipped, Phase 7 shipped anyway with one 1280├Ś720 screenshot of the first post.

## How to avoid it

1. **Add `oxfmt --check` to the Tier-1 quality gate and pre-push hook.** Either: (a) extend `vtz run lint` to invoke both oxlint and oxfmt so the existing `vtz test && vtz run typecheck && vtz run lint` chain in `CLAUDE.md` catches format, OR (b) add `packages/**/*.{ts,tsx,mdx}` to the pre-push hook's format check. Today's chain is lint-only; the phase-execution step in `.claude/rules/workflow.md` mentions oxfmt explicitly but autonomous agents are reading CLAUDE.md's shorter chain first. Aligning the two prevents this class of CI miss entirely.

2. **Make phase review a hard gate: no commit of the next phase until the review file has `Reviewer:` + `Findings:` + `Resolution:` filled.** The `local-phase-workflow.md` rule is aspirational; Phases 2ŌĆō7 demonstrated it can be skipped invisibly. Concrete fix: a pre-phase-commit check (local script or Git hook) that inspects `reviews/<feature>/phase-<N>.md` and blocks if the template placeholders (`_(pending)_`, `_To be completed by reviewer._`) are still present. Alternatively, change the phase-execution loop in `.claude/rules/workflow.md` to invoke the adversarial-review agent as a tool call with a typed success/failure result, not as a human-follow-up.

3. **Every "Deferred" line in a phase review becomes a filed GitHub issue before the phase closes.** Phase 4 deferred 5 items, Phase 5 deferred 2, Phase 6 deferred 4, Phase 7 flagged external-validator runs. Rule change in `.claude/rules/design-and-planning.md`: in the "Definition of Done ŌåÆ Phase" checklist, add `[ ] Every deferral documented in the phase review has a corresponding open GitHub issue linked inline.` Without an issue, the deferral isn't tracked; in three months it's untracked tech debt. The `feedback_phase_close_scope.md` memory already says "At phase close, issues for follow-ups are the artifact" ŌĆö formalizing it in the DoD closes the loop.

4. **Feed framework-contradiction discoveries back into the rules file in the same phase that discovered them.** Phase 2 found `domEffect` doesn't exist and `@vertz/icons` breaks SSR. The phase review noted it. `.claude/rules/ui-components.md` was not updated. Rule: any phase whose review includes a "Deviation" that contradicts a guidance rule MUST include, in the phase tasks, an edit to that rule file in the same commit range. Otherwise the next agent re-discovers the same wall.

5. **Add `reviews/` to `.gitignore`, OR move screenshots somewhere committable.** The `local-phase-workflow.md` rule says `reviews/` is an artifact; shipping it to main contradicts the rule. Either add `reviews/` to the repo root `.gitignore` and document that retros/PR bodies are the permanent record, OR amend the rule to say "reviews stay on the feature branch; screenshots go to `plans/<feature>/screenshots/`" and allow screenshots into main. The current "rule says no, merge says yes" state is the worst shape.

6. **"Launch checklist" as an explicit phase for features that ship hidden from the main navigation.** The blog is live but no one can find it. For any feature whose routes exist but aren't linked from nav/footer/sitemap, add a final "Launch" phase to the implementation plan: (1) wire nav/footer links, (2) dynamic sitemap entries, (3) external-validator runs against preview URL, (4) announcement plan. Alternatively, accept "hidden launch" as a pattern and file a dedicated `launch: <feature>` tracking issue that enumerates the visible-to-the-world steps. Either works; "silently live" is not a supported state.

7. **When a framework gap (like `vertz_browser_screenshot` viewport ignore) blocks a phase's QA, the phase does not close until either the gap is fixed OR the QA is re-done with a different tool.** Phase 2 shipped with 3 fake-viewport PNGs and a note saying "Phase 7 will re-run once #2949 ships." Phase 7 shipped. #2949 is still open. The QA debt rolled silently to the next feature. Rule change: phases cannot close with QA contingent on an un-merged fix ŌĆö either resolve the fix, or swap the QA strategy (e.g., Playwright / manual browser resize + screenshot) and rerun.

## Process changes adopted

1. **`oxfmt` joins the quality-gate chain.** Propose amending `CLAUDE.md` "Quality Gates" section to `vtz test && vtz run typecheck && vtz run lint && vtzx oxfmt --check packages/`, and updating the pre-push hook accordingly. Tracked as a follow-up issue.

2. **Phase reviews as a typed gate.** Propose the autonomous phase loop (`.claude/rules/workflow.md` Phase Execution step 4) invoke the staff-reviewer agent and block on a structured result (`{ blockers: [], shouldFix: [], approved: bool }`) rather than a markdown file. Tracked as a follow-up issue.

3. **Deferral ŌåÆ issue required in Definition of Done.** Propose amending `.claude/rules/design-and-planning.md` Phase DoD to require every "Deferred" item in a phase review to link to an open GitHub issue. Tracked as a follow-up issue.

4. **Stale-rule-feedback loop.** Propose amending `.claude/rules/design-and-planning.md` to require: "If a phase review contains a Deviation that contradicts a rule in `.claude/rules/*.md`, the phase tasks include editing that rule file." Tracked as a follow-up issue.

5. **`reviews/` policy decision needed.** Either `.gitignore` it or amend the rule to allow it; filing a discussion issue to pick one.

6. **"Launch" as an explicit phase for hidden-route features.** Propose amending `.claude/rules/phase-implementation-plans.md` to call this out.
